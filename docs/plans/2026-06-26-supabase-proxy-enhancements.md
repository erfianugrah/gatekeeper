# Supabase Proxy Enhancements: Rate Limiting + Scope Decomposition

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire rate limiting to the Supabase proxy (currently unconstrained) and decompose the overly-wide `projects` catch-all category into five focused scopes, giving operators the granularity needed for delegation use-cases like per-tenant billing overlays.

**Architecture:** Two independent subsystems — (1) rate limiting follows the identical pattern already used by S3 and CF proxy: a `TokenBucket` on the Durable Object, a `consumeXxx()` DO method, and a guard call in the router before upstream forwarding. (2) Scope decomposition adds five new `SupabaseCategory` values and their tail-prefix entries in `classify.ts`, plus fixes the `config/storage` inconsistency (storage config lands in `projects` catch-all while `storage/buckets` correctly lands in `storage`). Both changes are additive — no breaking changes to existing keys.

**Tech Stack:** TypeScript, Cloudflare Workers / Durable Objects, Hono, Vitest + `@cloudflare/vitest-pool-workers`, React (PolicyBuilder dashboard)

## Upstream API gaps (verified against /docs/supabase-api/)

Documented here so we don't plan work that depends on non-existent upstream endpoints.

| Customer ask | Management API reality | Gatekeeper can do |
|---|---|---|
| Per-project **storage quota** (total bytes) | ❌ No such field — `fileSizeLimit` is a per-file upload ceiling only | Gate `supabase:storage:write` to control who can change `fileSizeLimit` |
| **Connection limit** enforcement | ❌ `max_client_conn` on pgbouncer is read-only; `max_connections` is implicit in compute tier | Gate `supabase:billing:write` to control who can resize compute (pool size proxy) |
| **Billing-grade usage export** (bytes, compute-hours, egress) | ❌ Not in Management API — only request-count telemetry via analytics endpoints | Gate `supabase:analytics:read`; Gatekeeper's own `supabase_proxy_events` gives request-level tracking |
| **API rate limiting config** | ❌ Not exposed — `429`s from the Management API are Supabase-internal | Gatekeeper adds its own account-level token bucket (this plan) |
| **Disk resize** | ✅ `POST /v1/projects/{ref}/config/disk` sets IOPS/size_gb; `GET .../util` gives live bytes-used | New `supabase:disk:*` scope gates this separately from project lifecycle |
| **Pool size** (supavisor) | ✅ `PATCH /v1/projects/{ref}/config/database/pooler` sets `default_pool_size` | Already under `supabase:database:write` |

## What this enables for Customer

After this plan is implemented, a delegation key for a tenant end-user project can be constructed with:

```json
{
  "statements": [
    {
      "effect": "allow",
      "actions": [
        "supabase:analytics:read",
        "supabase:projects:read",
        "supabase:storage:read",
        "supabase:database:read",
        "supabase:billing:read"
      ],
      "resources": ["project:<their-ref>"]
    }
  ]
}
```

This grants monitoring + read access without exposing `billing:write` (compute resize), `disk:write` (storage resize), `networking:write` (ban/unblock), or any lifecycle ops (`projects:write` covers pause/restart/upgrade).

**Remaining gaps that require upstream Supabase changes** (not addressable in Gatekeeper):
- Total per-project storage quota (bytes ceiling) — `fileSizeLimit` is per-file only
- Billing-grade usage export (compute-hours, egress GB) — Management API only has request-count telemetry
- Per-project API rate limit config — not exposed in Management API

These are the feature requests worth raising with Supabase engineering / flagging on the upstream provider roadmap.
